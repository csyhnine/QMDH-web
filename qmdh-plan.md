# 设计院内部 AI 升级平台规划书

**文档版本**：v2.0  
**修订日期**：2026-04-15  
**适用范围**：设计院内部 AI 升级与统一接入平台  
**默认前提**：`1 名开发者 + 1 名业务审核人`

---

## 一页摘要

### 这是什么

这是一个面向设计院内部的 AI 升级平台，用来统一接入多家顶尖闭源模型 API，提供可复用工作流，沉淀任务与素材资产，并对成本、权限、日志和审计进行统一管理。

### 这不是什么

- 不是一期就覆盖设计院全部业务的“AI 操作系统”
- 不是对外商业化的 SaaS 产品
- 不是自研基础大模型项目
- 不是“所有数据都不上公有云”的全私有闭环方案

### 一期要做什么

- 建立统一模型接入层
- 建立统一工作流执行层
- 建立统一资产沉淀层
- 建立统一日志、成本、权限、审计能力
- 跑通 `3-5` 个高频示范工作流

### 一期不做什么

- 不承诺覆盖所有设计业务场景
- 不承诺建筑体块生成、图纸审查等探索型能力达到生产可用
- 不把 ROI 作为 12 个月硬验收指标

---

## 1. 项目定位

### 1.1 核心定义

本项目的本质不是“再做一个 AI 工具”，而是为设计院建立一个内部 AI 升级底座：

1. 统一不同模型供应商的调用方式
2. 把高频任务封装为可复用工作流
3. 把结果、提示词、素材和任务记录沉淀为内部资产
4. 让 AI 使用过程具备成本可见、权限可控、过程可审计

### 1.2 核心路线

平台默认采用“闭源模型 API 聚合 + 本地资产留存 + 工作流编排”的路线：

- 图像生成与编辑：接入顶尖闭源图像模型
- 文档与报告生成：接入通用闭源大模型
- 视频生成：按需接入视频模型
- 本地只保留必要的文件、任务、审计和资产管理能力

### 1.3 成功标准

项目成功不是“功能越多越好”，而是满足以下四件事：

- 设计院能稳定调用外部顶尖模型完成高频任务
- 使用结果能被复用，而不是每次从零开始
- 管理层能看到成本、活跃度、成功率和使用效果
- 在单开发者条件下，一期范围可以被稳定交付

## 2. 目标用户与一期范围

### 2.1 目标用户

| 用户角色 | 一期关注点 |
|------|------|
| 主创设计师 | 快速生成方案草图、效果图、汇报材料 |

| 管理层 | 查看平台活跃度、成本、产出效率、复用效果 |

### 2.2 一期业务目标

一期只解决“高频、明确、能复用”的问题，不追求大而全。

优先交付以下示范工作流：

1. 建筑/景观效果图生成
2. 局部遮罩重绘与改造
3. 效果图转 PPT/汇报页
4. 前期分析报告生成
5. 图片提示词反推

| 级别 | 类型 | 内容 |

| `P0` | 平台能力 | 统一日志、成本、权限、审计 |
| `P1` | 示范场景 | 效果图生成 |

## 4. 平台能力设计

### 4.1 P0-1：统一模型接入层

目标：把不同供应商 API 封装成统一调用接口。

一期必须具备：

- 统一的模型配置与供应商注册
- 统一的请求参数映射
- 统一的超时、重试、熔断、降级逻辑
- 统一的费用记录与调用日志
- 统一的任务状态回传

建议统一能力接口：

- `image.generate`
- `image.edit`
- `text.generate`
- `document.generate`
- `video.generate`

### 4.2 P0-2：统一工作流执行层

目标：把高频任务从“人工拼接多个工具”变成“平台内可复用工作流”。

一期能力边界：

- 工作流配置驱动
- 任务队列与异步执行
- 步骤级日志
- 手动重试与失败回放
- 输出物归档

一期不做：

- 复杂可视化拖拽编排器
- 多分支条件编排编辑器
- 通用 Agent 平台化市场

### 4.3 P0-3：统一资产沉淀层

目标：让每次任务都沉淀为下次可复用的资产。

一期沉淀对象：

- 原始输入文件
- 生成结果文件
- 提示词与参数
- 工作流版本
- 项目归属
- 标签与备注

一期核心能力：

- 资产上传与归档
- 标签与项目维度检索
- 历史结果复用
- 提示词复用

### 4.4 P0-4：统一日志、成本、权限、审计

目标：解决“用了什么、花了多少、谁用了、出了什么结果”。

一期必须有：

- 用户登录与基础角色权限
- 任务级调用日志
- 供应商级成本统计
- 项目级资产访问记录
- 导出审计记录

## 5. 一期示范工作流

### 5.1 工作流 1：效果图生成

输入：

- 白模截图、线稿或参考图
- 风格选项
- 补充描述

输出：

- `3` 张候选效果图
- 关联提示词、模型、参数、生成耗时、成本记录

### 5.2 工作流 2：局部重绘

输入：

- 原图
- 遮罩区域
- 局部修改描述

输出：

- 重绘结果
- 原图与结果对比
- 同步留存重绘参数

### 5.3 工作流 3：效果图转 PPT/汇报页

输入：

- 项目信息
- 效果图
- 关键说明文字

输出：

- 可编辑 PPTX
- HTML 汇报页
- 生成过程日志

### 5.4 工作流 4：前期分析报告生成

输入：

- 项目地址
- 项目类型
- 分析关键词

输出：

- 结构化报告草稿
- 引用来源清单
- 审核状态

### 5.5 工作流 5：图片提示词反推

输入：

- 图片

输出：

- 可编辑提示词
- 推荐风格标签
- 可直接复用到其他工作流

## 6. 技术架构

### 6.1 一期生产主路径

一期正式生产方案只保留一套主路径：

- 前端：React Web App
- 后端：FastAPI
- 任务队列：Redis
- 数据库：PostgreSQL
- 文件存储：对象存储或 NAS
- 工作流配置：YAML/数据库元数据

### 6.2 架构说明

```text
前端层
  React Web App
    ├─ 工作台
    ├─ 任务列表
    ├─ 资产库
    ├─ 工作流入口
    └─ 管理看板

服务层
  FastAPI
    ├─ 模型接入层
    ├─ 工作流执行层
    ├─ 资产管理层
    ├─ 权限与审计层
    └─ 报表与看板层

异步层
  Redis Queue / Worker

数据层
  PostgreSQL
    ├─ users
    ├─ projects
    ├─ tasks
    ├─ workflow_runs
    ├─ assets
    ├─ prompts
    ├─ provider_calls
    └─ audit_logs

文件层
  NAS / 对象存储
    ├─ images
    ├─ documents
    ├─ videos
    └─ exports

外部模型供应商
  ├─ 图像模型 API
  ├─ 文本模型 API
  └─ 视频模型 API
```

### 6.3 技术决策约束

- `PostgreSQL` 是一期正式生产数据库，不再使用“先 SQLite、后迁移”的双路线表述
- `SQLite` 仅允许用于单机原型或本地 PoC，不进入正式生产文档主路径
- 不把某一家模型供应商写死为唯一方案，供应商适配器必须可替换

## 7. 数据与安全策略

### 7.1 安全口径修正

本项目默认依赖闭源云 API，因此不能再使用“数据不上公有云”这类绝对表述。

正确口径应为：

- 数据分级
- 最小出域
- 脱敏优先
- 全链路留痕
- 供应商可配置

### 7.2 数据分级

| 数据级别 | 示例 | 默认策略 |
|------|------|------|
| `A 级` | 甲方原始资料、涉密图纸、合同材料 | 不允许直接出域，必须本地保存 |
| `B 级` | 项目方案草稿、效果图中间稿、内部文本 | 脱敏后可发送到指定供应商 |
| `C 级` | 通用参考图、公开样例、模板素材 | 可按工作流规则发送到供应商 |

### 7.3 出域规则

一期明确规则如下：

- `A 级` 数据默认不得发送到外部闭源模型
- `B 级` 数据需经过脱敏或裁剪后才能出域
- `C 级` 数据可根据工作流配置直接出域
- 每次出域必须记录：用户、项目、时间、供应商、模型、任务、数据级别

### 7.4 审计要求

必须记录以下审计字段：

- 谁发起了任务
- 任务属于哪个项目
- 调用了哪个供应商和模型
- 使用了什么工作流版本
- 输入输出文件 ID
- 消耗时长与费用
- 是否涉及出域

## 8. KPI 与验收口径

### 8.1 一期 KPI

一期不使用 `ROI ≥ 200%` 作为验收标准。

建议使用以下指标：

| 指标 | 定义 | 目标 | 统计周期 | 责任人 |
|------|------|------|------|------|
| 活跃使用率 | 周内至少完成 1 次任务的目标用户占比 | `>= 60%` | 周/月 | 业务审核人 |
| 任务成功率 | 最终成功完成的任务数 / 发起任务数 | `>= 90%` | 周/月 | 开发者 |
| 平均响应时长 | 从提交到可查看结果的平均耗时 | 按工作流单独统计 | 周/月 | 开发者 |
| 单任务平均成本 | 每类工作流的平均外部 API 成本 | 可见、可追踪 | 周/月 | 开发者 |
| 人工节省时长 | 与原人工流程对比的节省时长 | 选择 3 个高频场景核算 | 月 | 业务审核人 |
| 资产复用率 | 使用历史提示词/素材/结果的任务占比 | 持续提升 | 月 | 业务审核人 |
| 审计覆盖率 | 有完整日志和成本记录的任务占比 | `100%` | 周/月 | 开发者 |

### 8.2 验收方法

所有指标必须具备固定口径：

- 统计对象：哪些用户和哪些任务计入
- 计算公式：分子和分母清晰
- 抽样方式：自动统计或固定抽样
- 责任人：谁确认结果
- 数据来源：数据库报表或审计日志

### 8.3 不再使用的模糊指标

以下表达不再直接作为一级验收标准：

- 风格符合度 `80%`
- 区域一致性 `95%`
- 灯光匹配度 `70%`
- 报告差异化程度 `60%`
- 识别准确率 `85%`

这类指标只能作为内部参考项，除非补齐标注集、评分规则、判分责任人与抽样方法。

## 9. 里程碑与交付计划

### 9.1 Phase 1：底座上线期（0-3 个月）

目标：完成四项平台 `P0` 能力，并跑通 `3` 个示范工作流。

交付物：

- 模型接入层 v1
- 工作流执行层 v1
- 资产库 v1
- 日志、成本、审计 v1
- 工作流：效果图生成、局部重绘、提示词反推

验收标准：

- 目标用户可以通过统一入口发起任务
- 所有任务均有完整日志与成本记录
- 至少 `3` 个工作流可稳定运行

### 9.2 Phase 2：业务验证期（3-6 个月）

目标：补齐 `P1` 示范场景，完善项目视角管理和管理看板。

交付物：

- 工作流：PPT/汇报生成
- 工作流：前期分析报告生成
- 项目维度任务管理
- 成本统计与活跃度看板
- 数据分级与出域留痕规则落地

验收标准：

- `3-5` 个高频工作流稳定运行
- 管理层可查看使用量、成功率、成本与审计记录
- 脱敏与出域规则可检查、可复核

### 9.3 Phase 3：扩展与探索期（6-12 个月）

目标：在平台稳定前提下，有限探索 `P2/P3` 能力。

候选项：

- 夜景精控优化
- 品牌/IP 扩展工作流
- 建筑体块生成 PoC
- 图纸审查 PoC

说明：

- `P2/P3` 只做 PoC 或小范围试点
- 不进入一期硬承诺验收清单

## 10. 资源与交付约束

### 10.1 现实约束

当前默认交付模式：

- 开发：`1` 人
- 业务审核：`1` 人

因此文档中的所有承诺必须符合以下原则：

- 一期只做必要平台能力
- 只选择少量高频场景作为示范
- 先把“可用、可管、可追踪”做出来，再谈复杂智能化

### 10.2 范围控制原则

如果出现以下情况，必须收缩范围，而不是继续扩项：

- 新供应商接入带来大量适配工作
- 审计、权限、成本统计还未稳定
- 已有工作流成功率不达标
- 用户反馈集中在稳定性而非功能缺失

---

## 11. 成本与收益口径

### 11.1 成本口径

成本分为两类：

- 固定成本：开发时间、本地服务器、存储、基础设施
- 浮动成本：外部模型 API 调用费用

一期更重要的是建立“成本可视化能力”，而不是追求一开始就精确预测全年 ROI。

### 11.2 收益口径

一期建议按以下方式观察收益：

- 高频任务耗时下降
- 汇报材料产出速度提升
- 历史结果复用率提升
- 模型调用成本透明化
- 多供应商切换效率提升

### 11.3 ROI 使用原则

- ROI 只作为年度复盘指标
- 不作为 12 个月硬验收门槛
- 未建立稳定统计口径前，不在首页承诺具体高 ROI 数字

## 12. 风险与应对

| 风险 | 说明 | 应对策略 |
|------|------|------|
| 供应商价格或政策变化 | 闭源 API 价格、能力、可用性可能波动 | 适配器化封装，保留替代供应商 |
| 数据出域风险 | 闭源模型调用天然存在外部传输 | 执行分级、脱敏、审计与审批策略 |
| 单开发者负载过高 | 平台层和业务层都扩张时容易失控 | 固定 P0/P1 边界，探索项后置 |
| 工作流过多导致维护困难 | 每多一个工作流都增加维护成本 | 一期严格限制在 `3-5` 个 |
| 指标无法验收 | 只写目标不写口径会失真 | 所有 KPI 必须有定义、公式、责任人 |

本版文档针对上一版的关键问题进行了收敛：

## 14. 结论

这不是一个追求“大而全”的平台项目，而是一个面向设计院内部的 AI 升级底座项目。

一期最重要的不是做出最多功能，而是把以下四件事做稳：

1. 统一接入
2. 统一编排
3. 统一沉淀
4. 统一审计

只要这四件事稳定，再叠加 `3-5` 个高频示范工作流，平台就已经具备真实内部价值，也为后续探索更复杂的设计智能化能力打下了可靠基础。
