## Recommended Deploy Order (Learned The Hard Way)

When the release includes **new Alembic files** and **backend code changes**:

```bash
# 1. Pull as admin (never as root)
sudo -u admin git -C /www/wwwroot/qmdh-web fetch origin main
sudo -u admin git -C /www/wwwroot/qmdh-web pull origin main

# 2. Rebuild images FIRST (migration files live inside backend image)
cd /www/wwwroot/qmdh-web
docker compose build frontend backend worker

# 3. Run migrations with the NEW backend image
docker compose run --rm backend alembic upgrade head
docker compose run --rm backend alembic current   # expect repo head, e.g. g8h9i0j1k2l3

# 4. Restart
docker compose up -d frontend backend worker

# 5. Verify
docker compose ps
curl -fsS http://127.0.0.1:8080/api/v1/health?detail=full
curl -fsS https://cityusbdisk.cn/api/v1/health?detail=full
sudo -u admin git -C /www/wwwroot/qmdh-web rev-parse --short HEAD
```

### Why build must precede Alembic

`docker compose run --rm backend alembic ...` executes against the **backend Docker image**, not the host checkout alone.

If you pull new code but run Alembic **before** rebuild:

- The running image may not contain new files under `migrations/versions/`
- `alembic upgrade head` appears to succeed but stays on the old head
- After rebuild + restart, you must run migrations again

This happened during the v1.1.0 rollout: first `alembic current` showed `6f7a8b9c0d1e (head)` on the old image; only after rebuild did `6f7a8b9c0d1e -> g8h9i0j1k2l3` apply.

## Why The Deploy Took So Long

### 1. Full backend/worker rebuild (~47 min pip install)

Production had been on **7-day-old backend/worker images**. Prior Gemini fixes often used **hotpatch** (copy `.py` into containers) instead of `docker compose build backend`.

v1.1.0 required a full rebuild. Dockerfile step:

```dockerfile
RUN pip install --no-cache-dir -r requirements.txt
```

Observed duration on ECS: **`2799.7s` (~47 minutes)**.

`requirements.txt` did **not** change between `51aba1b` and `0090a2a`. The slow step was **Docker layer cache miss**, not new dependencies.

### 2. Parallel duplicate builds

Two deploy scripts overlapped, each running `docker compose build backend`. This wasted CPU/disk/network and made timing unpredictable.

**Rule:** never run two backend builds concurrently on the same host.

### 3. Frontend was fast; backend was not

| Step | Approx. time |
| --- | --- |
| `git pull` | 1–2 min |
| `build frontend` | ~20 s |
| `build backend` (pip uncached) | **~47 min** |
| `build worker` | bundled with backend Dockerfile |
| `alembic upgrade head` | seconds |
| `up -d` + health | ~1 min |

## Deploy Mode Comparison

| Mode | When to use | pip reinstall? | Typical time |
| --- | --- | --- | --- |
| **Hotpatch** | Urgent single-file backend fix; no new deps/migrations | No | minutes |
| **Rebuild frontend only** | UI/static/nginx changes | No | ~1 min |
| **Full rebuild backend/worker** | New backend code, migrations, dependency changes | Yes (if cache miss) | **minutes–~50 min** |
| **Pull prebuilt image (future)** | CI builds to ACR | No on server | minutes |

## Hotpatch vs Full Rebuild

**Hotpatch** (documented in `docs/archive/deploy-2026-06-22-gemini-hotpatch.log`):

- Copy changed files into running `backend` / `worker` containers
- `docker compose restart backend worker`
- Fast, but **image drift**: `docker inspect` image ≠ git HEAD code

**Full rebuild** (this rollout):

- Code + migrations baked into image
- Eliminates hotpatch drift
- Expensive when pip layer is uncached

After v1.1.0, production images and git `0090a2a` are aligned again.

## Git & Access Reminders

- `git pull` / `fetch` as **`admin`**: `sudo -u admin git -C /www/wwwroot/qmdh-web pull origin main`
- Running git as **`root`** fails (`Permission denied publickey`) — deploy key is on `admin`
- Fallback when GitHub fetch fails: **git bundle** upload (see `docs/continuity.md`)

## Docker Hub / Base Image Notes

Historical incidents: pulling `python:3.12-slim` from Docker Hub failed on this host.

During v1.1.0 build, `python:3.12-slim` was **CACHED** — pull was not the bottleneck this time.

If build fails at base image pull:

- Retry later, or
- Mirror base images to Alibaba ACR, or
- Use hotpatch for minimal backend fixes only

## Follow-Up Improvements

1. **CI → ACR**: build images off-server; production only `docker pull` + `up -d`
2. **Single deploy script**: **`scripts/deploy-production.sh`** — enforces pull → build → alembic → up; flock lock forbids parallel backend builds
3. **Document frontend-only vs full deploy** in release notes to avoid unnecessary backend rebuilds (see `docs/server-operations.md` Update Rule Of Thumb)
4. **Optional**: keep a warm Docker build cache on ECS; avoid `docker system prune` before releases
