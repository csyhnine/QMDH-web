## 本阶段目标

- 完成图像生成 MVP 1.0 主流程
- 保证前后端、任务执行、历史流和媒体访问可用
- 建立最小服务器部署基线

## 当前进展

- FastAPI 后端已提供项目、任务、工作流、图库、模板与媒体访问接口
- React 前端已形成图像生成工作台主界面
- 前端模板体系已开始收敛到建筑与景观效果图增强场景，并修复了模板弹层溢出问题
- `frontend/src/App.tsx` 已继续清理旧热门模板、模板保存死分支和旧本地兜底逻辑
- 前端工作台入口已收束到图像生成，避免继续暴露视频待开发分支
- 前端已移除单项工作流选择菜单，当前提交链路固定走图像生成工作流
- 前端提交任务后会强制刷新历史流并重新定位最新任务，避免被后台轮询状态跳过
- 参考图上传已补充 input 重置，支持失败后或同文件重复选择时重新触发上传
- 前端数据刷新已增加最新请求写入保护，避免并发响应覆盖较新的历史流状态
- 历史流已区分项目无历史与筛选无结果，筛选为空时仍保留筛选栏供用户恢复
- 真实图像 provider 已接通 ModelScope 日配额方案
- 提示词模板已从前端本地存储升级为后端持久化
- 参考图已支持真实上传并保存到后端媒体目录
- 参考图已通过 `caption_prompt` 模式进入真实生成链路：先读图生成参考说明，再拼入文生图 prompt
- 已加入 MVP 最小 token 认证与项目级访问过滤，任务和模板不再信任前端传入的 `user_name`
- 前端会跟随认证后的可见项目列表切换当前项目，避免默认项目不在授权范围时继续提交
- 前端切换项目时会同步项目分级，避免提交任务沿用旧项目的 classification
- 已补充 Docker 部署基线与部署说明
- 已补充 Windows 本地一键开发启动脚本，可同时拉起前端与后端开发服务
- 已新增独立后台模型管理入口 `/admin/models`，可维护 provider、model、base URL、API key、能力、启停状态和参考图模式
- 模型管理已从设计师工作台侧栏拆出，并通过 `admin / owner / ops` 角色限制访问
- 后端 provider 注册表已支持数据库配置，任务创建和执行都会读取后台保存的真实 provider profile
- 当存在 ModelScope token 时，设计师模型列表会自动出现 5 个真实魔搭 provider：`MAILAND/majicflus_v1`、`Qwen/Qwen-Image-2512`、`Tongyi-MAI/Z-Image`、`Tongyi-MAI/Z-Image-Turbo`、`FireRedTeam/FireRed-Image-Edit-1.1`
- 设计师页面已不再显示 `jimeng`、`nano_banana` 等模拟 provider

## 风险与阻塞

- 当前参考图接入是“语义参考”方案，还不是直接 `img2img / image.edit` 能力
- 当前用户边界是 MVP token 认证，不是完整账号、会话、密码或 SSO 体系
- `frontend/src/App.tsx` 还有历史重构遗留，后续需要收口清理
- 当前热门模板已切到更贴近建筑/景观业务的方向，但模板配置仍在前端代码里，后续可考虑继续后端化
- 当前 API key 采用数据库明文保存并在前端脱敏展示，生产环境仍需要补密钥加密、轮换和审计策略
- 新增的魔搭模型仍需逐个实测请求/返回格式和建筑提示词跟随效果；部分 API-Inference 模型可能需要专用 adapter

## 最近决策

- MVP 1.0 阶段优先保证“生图功能可用 + 可部署”，不继续扩散到视频模块
- 先以 ModelScope 作为真实生图 provider 验证链路
- 模板与参考图都要从“前端临时状态”升级到后端真实数据
- MVP 1.0 先用配置型 token 认证建立可信执行人和项目访问边界
- 模型与 key 管理先做 MVP 后台配置，不把真实 key 暴露到前端，也不再只依赖 `.env` 才能切换 provider
