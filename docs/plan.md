# Plan

## 1. Project Goal
QMDH-web 的目标是为设计院内部搭建一个可持续演进的 AI 升级平台原型，统一接入多类模型能力，并通过工作流、任务记录和资产沉淀提升高频设计场景的执行效率。

当前阶段的现实目标不是做“大而全”的平台，而是先把图像生成等高频场景打磨成可复用的最小可行产品，同时补齐后续多 agent 接手开发所需的协作文档体系。

---

## 2. Success Criteria

项目达到当前阶段成功，至少应满足以下标准：

- 新 agent 可通过仓库文档和 Git 状态快速接手，而不需要重复解释背景。
- 前后端主流程可以在本地跑通，至少包含工作流查询、任务创建、任务状态查看和资产浏览。
- 图像生成模块具备最小可用闭环：发起任务、查看结果摘要、复用图库资产。
- 代码、任务状态、交接记录和关键决策能够保持一致，不依赖单次会话上下文。
- 后续迭代具备清晰的优先级：先补真实能力与可靠性，再扩展功能面。

---

## 3. Current Stage

- 当前阶段：MVP
- 当前阶段目标：
  - 稳定交付图像生成模块的最小可行版本
  - 将项目从“可运行原型”推进到“可持续接手开发”的状态
  - 建立正式协作文档，支撑多 agent 接力开发
- 当前阶段非目标：
  - 一次性覆盖全部设计业务场景
  - 在本阶段内实现复杂权限、审批和全量文件治理
  - 把视频、报告、图库等所有模块同时打磨到生产级

---

## 4. Milestones

### Milestone 1: 建立协作事实层
- 目标：
  - 建立 `protocol / plan / architecture / decisions / tasks / handoff / review`
  - 明确当前工作区、当前主目标和接手流程
- 完成标准：
  - 新 agent 可以在不依赖聊天记录的情况下完成接手摘要

### Milestone 2: 稳定生图 MVP
- 目标：
  - 收束前端为专用图像生成工作台
  - 保持工作流、任务、模型、资产的最小闭环可用
  - 明确 UI 与后端当前真实能力边界
- 完成标准：
  - 本地可以完成“提交生图任务 -> 查看最近任务 -> 浏览图库资产”的完整操作链

### Milestone 3: 接入真实图像能力
- 目标：
  - 用至少一个真实图像模型供应商替换模拟适配器
  - 明确出参与结果返回结构
- 完成标准：
  - `image-generate` 任务返回真实产物引用，而不只是模拟结果摘要

### Milestone 4: 资产闭环与基础治理
- 目标：
  - 打通任务结果入图库
  - 增加基础认证、项目级访问控制和更清晰的错误链路
- 完成标准：
  - 已完成任务可以沉淀为可查询资产
  - 基础身份和访问边界不再完全依赖前端传参

---

## 5. Risks

- 当前后端仍以模拟 provider 为主，容易让前端交互看起来比真实能力更成熟。
- 工作区当前存在未提交前端改动，如果不及时整理，后续接手容易混淆“稳定状态”和“进行中状态”。
- 项目当前没有测试体系、迁移体系和 CI，任何继续扩展都可能放大回归风险。
- 安全边界尚未建立，用户身份和任务归属在现状下不可完全信任。
- `docs/ai-agent-project-docs/` 仍保留为模板目录，如不明确说明，后续 agent 可能误将模板当正式事实源。

---

## 6. Near-Term Priorities

1. 整理并确认当前生图工作台改动，形成一个清晰可交接的前端基线。
2. 接入至少一个真实图像模型 provider，替换 `task_executor.py` 中的模拟执行分支。
3. 设计真实结果结构与资产入库流程，打通从任务到图库的链路。
4. 引入最小测试与验证路径，至少覆盖任务创建、provider 兼容性和前端构建。
5. 开始补基础认证与项目级访问控制，防止身份和审计继续完全依赖前端传参。
