# QMDH 2.0 升级预备方案

Last updated: `2026-05-16`

## Purpose

本文档用于定义 QMDH 2.0 的升级方向，以及 1.0 后续迭代应如何为该方向预留地基。

当前不把“智能体 / 工作流平台”作为立即落地目标，而是把它视为未来升级路线。  
1.0 继续服务现有设计生产闭环，但后续所有较大改动都应尽量兼容未来的多步骤工作流、研究链路、智能协作和可追踪执行模型。

## 1. Version Positioning

### QMDH 1.0

定位：

- 稳定可用的设计生产平台
- 重点聚焦生成、灵感、项目、模型管理、内容沉淀和数据保全

当前目标：

- 把已有 MVP 持续打磨成可运营、可部署、可保全数据的内部平台
- 不在当前阶段直接扩展成通用智能体平台

### QMDH 2.0

定位：

- 研究型 / 协作型智能工作流平台
- 在 1.0 之上增加多步骤编排、外网研究、工件沉淀、可追踪执行和半自动协作能力

首个方向默认保持为：

- 设计师优先
- 半自动执行
- 研究型工作流优先

## 2. Planning Rule For 1.0

1.0 后续规划应遵守以下原则：

1. 不直接上线完整智能体系统
2. 不为了“看起来更先进”提前引入过重编排框架
3. 新增能力必须尽量避免阻断 2.0
4. 只有那些同时服务 1.0 当前需求、又能作为 2.0 地基的改进，才优先推进

## 3. 1.0 应优先补的 2.0 地基

### 3.1 数据分层清晰化

后续所有新功能在设计时都必须先回答：

- 这是代码、配置、业务数据、运行态，还是初始化内容？
- 这是长期历史，还是运行中间态？
- 这类数据在升级后应保留、可重建，还是可丢弃？

目标：

- 为未来 2.0 的运行记录、研究工件、工作流状态预留清晰归属

### 3.2 任务模型向“可编排”方向兼容

当前 `Workflow + Task` 保留，不推翻。  
但后续 1.0 内新增任何复杂功能时，应避免继续把逻辑硬塞进单一 `Task.payload / result`。

要求：

- 单步任务仍可运行
- 状态、步骤、产物、来源、审计信息开始结构化
- 新增字段或接口尽量为未来多步骤执行留余地

### 3.3 媒体存储向“工件存储”演进

灵感图、生成图、上传图、参考图、研究快照、摘要文件等，应逐步统一为可追踪资产 / 工件。

要求：

- 1.0 中凡是新增的图片、文本快照、摘要结果，都尽量走统一存储接口
- 平台展示资源尽量走受控存储，而不是长期依赖第三方热链

### 3.4 项目维度沉淀能力增强

“项目”应逐步从纯生成容器演进成内容沉淀容器。

新增能力优先考虑：

- 项目内灵感沉淀
- 项目内生成历史归档
- 项目内聊天 / 讨论记录归档
- 项目内标签 / 结论结构化

### 3.5 观测与审计能力继续增强

1.0 后续每新增一类耗时操作、模型调用、内容入库，都要继续补：

- 日志
- 来源
- 状态
- 失败原因
- 关键操作者

原则：

- 先为单步任务可追踪打牢
- 再为 2.0 的多步骤工作流提供扩展空间

### 3.6 运维与迁移能力前置

所有 1.0 后续升级都必须继续遵守：

- migration 升级结构
- 不清空线上卷
- 不更换加密 key
- 备份数据库和媒体

原因：

- 2.0 未来一旦引入工作流运行记录和研究资产，数据只会更复杂，不会更简单

## 4. 当前明确不做的 2.0 内容

以下内容在当前阶段只停留在方向设计与架构准备，不作为立即落地目标：

- 通用 Agent 平台
- 低代码工作流编排器
- 多角色审批流系统
- 全自动后台自治智能体
- 复杂多智能体协作框架
- 把生成、研究、交付能力一次性编排成总工作流

## 5. 2.0 Compatibility Check

从现在开始，1.0 后续每个较大的需求在立项或评审时都必须补一轮 “2.0 兼容性检查”：

1. 这个改动是否会阻碍未来多步骤工作流？
2. 这个数据结构是否还能扩展来源、步骤、工件、状态？
3. 这个功能结果是否能沉淀到项目维度？
4. 这个执行过程是否可追踪、可恢复、可审计？
5. 这个能力以后是否能被研究型 / 协作型智能体复用？

这条检查不要求当前实现 2.0，只要求不把路封死。

## 6. Stage Model

### Stage A: 1.0 稳定化

目标：

- 修复现有体验问题
- 完善部署、数据、备份、灵感库、模型管理、账号体系
- 加强项目沉淀和资产可控性

### Stage B: 1.0 向 2.0 过渡

目标：

- 补可编排兼容的数据模型边界
- 补更强的审计 / 状态 / 工件沉淀能力
- 明确研究型工作流的信息架构
- 形成 2.0 技术设计输入

### Stage C: 2.0 正式立项

目标：

- 在已有地基上新增真正的编排层
- 再做研究型智能体和半自动工作流产品化

## 7. What This Means For Future Agents

后续 agent 在处理任何中大型需求时，应先同步阅读：

1. `docs/roadmap-2.0-prep.md`
2. `docs/data-governance.md`
3. `docs/architecture.md`
4. `docs/decisions.md`

并确认：

- 当前需求是 1.0 稳定化事项，还是 2.0 地基事项
- 如果只服务 1.0 当前闭环，是否会制造未来 2.0 的结构性负担
- 如果已经影响数据归属、存储层或项目沉淀方式，是否已同步更新文档
