# Takeover Prompt

你现在是一个新接手该项目的执行 agent。  
不要假设你继承了之前的聊天上下文。你唯一应该继承的是仓库中的代码、文档和 Git 状态。

你的任务是按以下顺序完成接管，然后继续开发：

## 一、先核对本地项目状态与 Git 状态
先检查以下内容：
1. 本地工作区是否干净
2. 当前所在分支
3. 本地是否落后于远端
4. 是否存在未提交改动
5. 最近提交记录是否与 `docs/handoff.md`、`docs/tasks.md` 一致

如果发现代码状态、文档状态、Git 提交记录三者不一致，请先指出差异，不要直接继续开发。

## 二、阅读项目上下文文档
你必须优先阅读并整理以下内容：
1. `docs/protocol.md`
2. `docs/tasks.md`
3. `docs/handoff.md`
4. `docs/plan.md`
5. `docs/architecture.md`
6. `docs/decisions.md`

阅读后，请输出一份简明接管总结，至少包括：
- 项目当前目标
- 当前开发阶段
- 当前最高优先级任务
- 受影响模块
- 需要遵守的限制
- 上一个 agent 留下的未完成事项
- 当前最合理的下一步

## 三、核对上下文和代码是否一致
你需要判断：
- 文档描述的系统结构是否与代码一致
- `tasks.md` 中的任务状态是否真实
- `handoff.md` 记录的修改是否已经体现在代码中
- 是否存在“文档说完成了，但代码没完成”或“代码改了，但文档没更新”的情况

如果不一致，请先列出不一致项，并提出修正建议，再继续开发。

## 四、继承系统后继续执行
完成接管后，再根据 `tasks.md` 中最高优先级任务继续开发。

要求：
- 只处理当前优先级最高且边界清晰的任务
- 不要擅自扩大任务范围
- 不要重构无关模块
- 修改后同步更新：
  - `docs/tasks.md`
  - `docs/handoff.md`
  - 必要时更新 `docs/architecture.md` / `docs/decisions.md`

## 五、输出形式要求
请按以下格式输出：

### 1. 接管检查结果
- Git 状态
- 本地与远端一致性
- 文档与代码一致性
- 是否可以继续开发

### 2. 项目上下文摘要
- 当前目标
- 当前任务
- 当前风险
- 下一步行动

### 3. 开发执行结果
- 修改了哪些文件
- 完成了什么
- 还有什么未完成
- 文档是否已同步更新

### 4. 交接摘要
面向下一个 agent 写一段 handoff，总结本次工作结果和注意事项

## 六、执行原则
- 仓库内容高于聊天内容
- 代码事实高于文档表述
- 若发现冲突，先对齐状态，再继续开发
- 每次执行都要留下可被下一个 agent 接手的明确痕迹
